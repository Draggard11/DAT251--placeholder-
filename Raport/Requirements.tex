
\section{Requirements}

\subsection{Overview}
Based on the pre-study, a set of functional and non-functional requirements were derived
to address the challenges related to motivation, engagement, and learning.

\subsection{Functional Requirements}
A brief summary of the functional requirements is presented here.
Full user stories are included in section of sprint 1~\ref{User-Stories}
The requirements are derived from user needs identified through personas
and interview insights.
For example, Fiona requires support for independent learning,
while Elias requires structure and guidance.

\begin{itemize}
    \item Students should be able to track their progress without relying solely on grades
    \item Students should be able to ask anonymous questions
    \item Students should be able to participate in study groups
    \item The system should provide feedback on participation
    \item The system should support gamification elements such as streaks and XP to increase engagement and motivation
\end{itemize}

\subsection{Non-functional Requirements}
\begin{itemize}
    \item Usability: The system must be intuitive and easy to use
    \begin{itemize}
        \footnotesize
        \item[] To compete with existing solutions to the problem, we should avoid friction that could lead to users preferring alternatives.
    \end{itemize}
    \item Accessibility: The system should work on common devices
    \begin{itemize}
        \footnotesize
        \item[] Similar to the above, there are potential users without access to smartphones or who may be unwilling to download additional applications, which the system should accommodate.
    \end{itemize}
    \item Scalability: The system should support many users
    \begin{itemize}
        \footnotesize
        \item[] Some functionality of the system is expected to benefit from a larger user base, such as group-creators, competitions, and shared user-created quizzes.
    \end{itemize}
    \item Maintainability: The system should be modular and extensible
    \begin{itemize}
        \footnotesize
        \item[] Especially during prototyping/feedback cycles, it is important to easily accommodate for new feature creation/deletion/adjustments
    \end{itemize}
\end{itemize}

\subsection{Sustainability Considerations}

At this stage of the project, sustainability considerations are inherently limited,
as no concrete solution has yet been defined.
Nevertheless, several relevant aspects have been identified.

From a technical perspective, the solution must align with users’ existing knowledge and access to technology, 
such as personal computers and mobile devices. 
This implies that any future system should be easy to use and accessible within realistic technical constraints.

From a social perspective, the project focuses on improving motivation, engagement, and encouragement within educational contexts. 
Addressing these issues has the potential to positively impact users’ well being and overall learning experience.

Regarding economic sustainability, no budget has been defined at this stage. 
The project will explore possibilities for making the solution freely available. 
No physical production is planned.

Environmental considerations include limiting or avoiding excessive use of resource intensive AI technologies, as high computational usage may have negative environmental impacts.

To evaluate the broader impact of our system, we applied principles from the sustainability
assessment framework presented in\cite{SAF-Toolkit}.
This framework emphasizes analyzing software systems across multiple dimensions,
including technical, environmental, economic, and social aspects.

Applying the framework resulted in the development of
several artefacts that support analysis of the system’s
broader impact.

First, a systemic impact map was created to visualize how the system influences different
stakeholders and resources.
This made it possible to identify both positive effects,
such as increased student motivation and social interaction, and potential negative effects,
such as increased server load and energy consumption.

Second, a set of quality attributes was defined, linking system features to sustainability
dimensions and measurable indicators.
This provided a structured way to reason about trade-offs,
for example between usability and performance, or between richer functionality and environmental impact.

Finally, key performance indicators (KPIs) were established to evaluate whether the system
achieves its intended goals over time.
These include indicators related to student engagement,
participation in study groups, and system resource usage,
enabling continuous evaluation of both user impact and system performance.

Overall, the use of the SAF framework contributed to a more holistic understanding of the system.
It supported more informed design decisions and highlighted areas where future improvements are needed,
particularly in balancing scalability, performance, and environmental impact as the system grows.

However, the assessment is limited by the early stage of development,
and several assumptions were necessary due to the lack of real usage data.
Future iterations should validate these assumptions through empirical evaluation.
